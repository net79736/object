## 문제 1-1: 주문 처리 시스템
```java
public class OrderService {
    public void processOrder(Order order) {
        Customer customer = order.getCustomer();
        
        // 고객의 등급 확인
        if (customer.getGrade().getLevel() >= 3) {
            // VIP 고객은 배송비 무료
            order.getDelivery().setFee(0);
        }
        
        // 재고 확인
        for (OrderItem item : order.getItems()) {
            Product product = item.getProduct();
            if (product.getStock().getQuantity() < item.getQuantity()) {
                throw new OutOfStockException();
            }
            product.getStock().decrease(item.getQuantity());
        }
        
        // 결제 처리
        PaymentMethod payment = customer.getPaymentMethods().get(0);
        payment.charge(order.getTotalAmount());
    }
}
```
질문:

이 코드가 위반하는 원칙들을 모두 찾으시오
각 위반이 왜 문제가 되는지 설명하시오
향후 발생할 수 있는 구체적인 문제 상황을 예측하시오

위반 원칙들:

디미터 법칙 위반 (심각)

customer.getGrade().getLevel() - 2단계 체이닝
product.getStock().getQuantity() - 2단계 체이닝
customer.getPaymentMethods().get(0) - 컬렉션 내부 접근

묻지말고 시켜라 위반

Customer에게 등급을 물어본 후 OrderService가 배송비 결정
Product에게 재고를 물어본 후 OrderService가 감소 처리

높은 결합도

OrderService가 Customer, Grade, Order, Delivery, Product, Stock, PaymentMethod의 내부 구조를 모두 알아야 함

발생 가능한 문제:

Grade 클래스가 level 대신 tierName으로 변경되면 → OrderService 수정 필요
Stock을 별도 클래스가 아닌 Product 내부 필드로 변경하면 → OrderService 수정 필요
결제 수단 선택 로직이 "첫 번째"에서 "기본 설정된 것"으로 변경되면 → OrderService 수정 필요
총 3개 클래스 변경 시 OrderService도 함께 변경 (높은 결합도)

개선 방향:

```java
public class OrderService {
    public void processOrder(Order order) {
        order.applyVipBenefits();  // Customer가 알아서 판단
        order.reserveStock();      // Product가 알아서 처리
        order.processPayment();    // Customer가 알아서 결제수단 선택
    }
}
```
